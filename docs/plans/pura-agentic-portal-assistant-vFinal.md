# Pura: Fully Agentic Portal Assistant (End-State Plan)

## Goal (non-negotiable)
Pura becomes a *fully agentic* in-portal AI employee:
- If a user can do it by clicking in the portal UI, Pura can do it via portal APIs **without requiring the user to paste IDs**.
- Pura executes when instructed, asks only for truly missing details, and never replies with “here’s how to do it manually” unless the user explicitly asks “how do I…”.
- The chat experience includes a ChatGPT-style **embedded “canvas”** that shows what Pura is working on (automation editor, funnel editor, campaign, contact, etc.), is resizable, and appears only when relevant.

This document is the source of truth for “done”. Implementation work is not complete until every requirement here is met and shipped.

---

## Product requirements

### 1) Command-first behavior
- **Imperative = execute**: Messages like “Create a funnel called X”, “Tag Chester VIP”, “Start campaign”, “Fix this automation” should trigger execution.
- **Questions = explain**: Messages like “How do I create a funnel?” should return instructions.
- **Correction handling**: Follow-ups like “No, remove tag A and use B”, “Actually make it weekly”, “Undo that” must be interpreted in context and executed.
- **Never ask for IDs**: If an action needs an ID, Pura resolves it from names/labels/URLs/recent context.

### 2) Parity with the portal
- Any portal feature that has an API endpoint / action can be planned and executed by Pura.
- “Parity” includes create/update/delete/enable/disable, as well as cross-service workflows (e.g., “Create funnel + connect to booking + add follow-up step”).

### 3) No confusing or meta actions
- Pura must never propose internal plumbing actions to the user (e.g., `ai_chat.*`, “create a new chat thread”, “flush scheduling”, “cron run”).
- No “Action finished” / duplicate “Done.” messages.
- Results must be explicit and stateful: what changed, on what entity, and what’s next.

### 4) Embedded canvas (ChatGPT-style)
- UI becomes a 2-pane system:
  - Left: threads list.
  - Middle: chat.
  - Right: **canvas panel** that can show an embedded portal surface relevant to the current work.
- Canvas requirements:
  - Appears only when Pura is actively working on something that has a meaningful UI representation.
  - Has an **X** to close; closing does not delete work.
  - Resizable horizontally by dragging.
  - Canvas can be replaced when Pura starts a new “work item” (with a clear transition).
  - Canvas can navigate within embedded content and maintain state.
  - Links generated by Pura that relate to executed work open in canvas (default), with an “open in new tab” affordance.

### 5) Resizable chat width
- The chat panel itself is resizable (or layout presets like compact / balanced / wide).

## System design (end-state)

### A) Core concept: Work Items
Pura operates on explicit **Work Items** rather than ad-hoc replies.

A Work Item represents an objective (e.g., “Create funnel”, “Fix automation”, “Tag contact”) and includes:
- `workItemId`
- `goal` (human-readable)
- `status`: `planning | executing | waiting_for_user | done | failed`
- `context`: inferred entities (contact, campaign, funnel, automation, booking calendar, etc.)
- `artifacts`: URLs/IDs and UI routes relevant to display in canvas
- `executionTrace`: actions performed, inputs, outputs, errors

Persist Work Items per thread so follow-ups can refer to them.

### B) Three-layer agent architecture

#### 1) Intent & Planning (LLM → JSON)
A single “planner” call converts the user message + recent context into a structured plan.

**Planner output schema (conceptual)**
```json
{
  "mode": "execute" | "clarify" | "explain" | "noop",
  "workItem": { "title": "...", "domain": "contacts|funnels|automations|..." },
  "entities": { "contactHint": "Chester", "funnelName": "...", "url": "..." },
  "steps": [
    {
      "actionKey": "contacts.tags.remove",
      "args": { "contactId": {"$ref":"contact"}, "tagId": {"$ref":"tag:as"} },
      "notes": "...",
      "ui": { "open": "/portal/app/people/contacts?focus=..." }
    }
  ],
  "clarifyingQuestion": "... (only if mode=clarify)",
  "explanation": "... (only if mode=explain)"
}
```
Key points:
- Planner can emit multi-step plans (remove + create + add).
- Planner never invents IDs.
- Planner references entities by *handles* (like `$ref`) which are resolved by the resolver layer.

#### 2) Resolution (deterministic)
A deterministic **Resolver** layer turns handles into real IDs.

Resolution strategies include:
- From current page URL (if user is in a funnel editor page, resolve funnelId from URL params)
- From thread context / last work item artifacts
- From name → lookup queries (e.g., contact by email/phone/nameKey)
- From “create if missing” rules (tags, folders, drafts, etc.)
- From cached lists when safe (e.g., last 50 funnels)

If resolution is ambiguous, the resolver returns a *single targeted question*:
- “I found 3 contacts named Chester—what’s the email or phone?”

#### 3) Execution (whitelisted actions)
An Executor runs steps via the existing whitelisted portal action system.

Executor requirements:
- Executes steps sequentially with partial success reporting.
- Uses idempotent APIs or pre-checks to avoid duplicates.
- Produces a **clean user-facing summary** and a detailed internal trace.
- On failure, returns:
  - what it attempted,
  - why it failed,
  - what info is needed (if any),
  - a suggested next step.

### C) Safety, permissions, and confirmations
- “Auto-execute” is default for low-risk changes; confirmations required for destructive operations.
- Confirmations are in-app modals (no native browser confirm).
- Confirmation policy is deterministic (not model-decided):
  - delete, cancel subscription, disconnect integrations, bulk sends → confirm
  - read-only, drafts, previews → no confirm

### D) Action registry + UI routing
Create a formal registry that pairs every action with:
- Arg schema (already exists via Zod)
- Resolver hints (how to find IDs)
- UI route mapping (where to open in canvas after execution)
- Risk level (confirm policy)

This registry is what makes “parity” real.

### E) “No instructions unless asked” rule
Before falling back to support-chat instructions:
1. Run planner.
2. If planner says execute → resolve → execute.
3. If planner says clarify → ask 1 question.
4. Only if planner says explain (or user asks how) → explain.

This prevents the exact failure mode you’re seeing (Pura giving manual steps after an imperative).

## UI plan (end-state)

### 1) Layout
- Add a resizable split layout:
  - Left: threads list
  - Center: chat
  - Right: canvas (optional)

### 2) Canvas implementation
- Canvas is an embedded portal surface:
  - Option A: `iframe` with same-origin route
  - Option B (preferred): internal React “portal view” that mounts existing pages/components

Canvas API:
- `openCanvas({ title, url, workItemId })`
- `closeCanvas()`
- `replaceCanvas(...)`
- `setCanvasWidth(px)` persisted per user

### 3) Link handling
- Links that refer to portal work open in canvas by default.
- External links open in a new tab.

### 4) Message rendering
- One assistant message per operation summary.
- No transient duplicates; eliminate separate “autoActionMessage” rendering as a second chat message.

## Engineering plan (how we get to the end-state)

### Phase 1 — Foundations (must-haves)
1. Planner JSON endpoint
   - Implement a unified planner call used by all domains.
   - Strict JSON schema validation; refuse invalid outputs.
2. Resolver framework
   - Standard interface: `resolve(handle, context) -> { id | clarifyQuestion }`.
3. Executor improvements
   - Transaction-like execution trace.
   - Clean result formatting (no generic “Done.”).
4. Kill meta actions
   - Ensure `ai_chat.*` never appears in model action lists and never renders as user-visible actions.

### Phase 2 — Domain coverage (parity)
For each service domain (funnels, automations, inbox, booking, newsletter, reviews, billing, etc.):
- Define resolver strategies for common handles.
- Add post-action UI route mapping so canvas opens to the right place.
- Add ambiguity handling.

### Phase 3 — Canvas + resizable layout
- Add canvas panel + resizing.
- Add “open result in canvas” behaviors.
- Add work item transitions.

### Phase 4 — Quality bar
- Automated tests for:
  - planner schema
  - resolver ambiguity
  - executor summary
  - regression coverage for previous failures (duplicate messages, wrong tag names, etc.)
- Observability:
  - store execution traces
  - correlate planner → resolver → executor IDs

---

## Definition of done
Pura is “done” when:
- Users can issue natural language commands across **all portal services** without IDs.
- Pura executes reliably, asks targeted questions only when necessary, and never defaults to manual UI instructions for imperative commands.
- Canvas panel behaves exactly as described (appears only when relevant, resizable, closable, opens the thing being worked on).
- No internal/meta actions appear.
- No duplicate “Done.”-style responses.
- Logs/traces exist for every execution.

---

## Notes on current pain points (to be eliminated)
- Duplicate “Done.” messages: indicates the system is emitting multiple UI-visible outputs for a single execution path.
- “Remove A and use B” mis-execution: requires multi-step planning + context-aware follow-ups.
- “Then make one” should resolve via Work Item context and planner inference.
